# Guest Access API Contract

This file documents the locked contract for guest access workflow endpoints used by the frontend.

OpenAPI remains available at `/docs` and `/redoc`.

## Workflow Rules

- One `zone_id` has one active reusable primary guest token.
- Primary guest token does not auto-expire (`expires_at` is always `null`).
- Guest request approval/rejection lifecycle drives `PERMISSION` system events.
- Manual `PERMISSION` sends are rejected for member and guest compose flows.
- Guest messaging is `CHAT` only after successful guest login.

## Canonical Endpoints

- `GET /api/access/qr-tokens/primary?zone_id={zone_id}`: get-or-create active primary token.
- `POST /api/access/qr-tokens/primary/rotate`: revoke active primary token and issue a new one.
- `POST /api/access/permission`: anonymous guest submit request.
- `GET /api/access/session/{guest_id}?zone_id={zone_id}`: poll `PENDING | APPROVED | REJECTED`.
- `GET /api/access/guest-requests?zone_id={zone_id}`: admin list requests by zone.
- `POST /message-feature/access/guest-requests/{requestId}/approve?zone_id={zone_id}`: approve request (zone_id required).
- `POST /message-feature/access/guest-requests/{requestId}/reject?zone_id={zone_id}`: reject request (zone_id required).

## Backward Compatibility

- `POST /api/access/qr-tokens` and `GET /api/access/qr-tokens` remain supported.
- Primary mode via old create endpoint (`is_primary=true`) keeps non-expiring semantics.
- If primary mode receives `expires_at` or `expires_in_hours`, backend returns clear validation error `PRIMARY_TOKEN_EXPIRY_NOT_ALLOWED`.

## Automatic Message Policy

`PERMISSION` is generated by server only for:

1. guest request submitted
2. admin approved
3. admin rejected

Manual compose requests with `PERMISSION` return `PERMISSION_MANUAL_DISABLED`.

## Guest Role Boundaries

Guest can:

- read own guest profile/session payload
- read allowed zone dashboard payload
- read allowed member peers for messaging
- read guest thread history
- post `CHAT` only to authorized host/admin peers

Guest cannot:

- create/update/delete zones or members
- send non-`CHAT` message types
- send manual `PERMISSION` events

## Error Codes

Minimum stable codes for frontend handling:

- `INVALID_GUEST_TOKEN`
- `TOKEN_ZONE_MISMATCH`
- `PERMISSION_MANUAL_DISABLED`
- `GUEST_MESSAGE_TYPE_NOT_ALLOWED`
- `GUEST_NOT_AUTHORIZED_FOR_ZONE`
- `ACCESS_REQUEST_NOT_FOUND`
